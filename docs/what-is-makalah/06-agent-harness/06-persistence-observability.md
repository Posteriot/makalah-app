## 1. Filosofi: Observability as Autonomy Enabler

Karena agen memiliki otonomi untuk menentukan langkah penalarannya sendiri (TAO cycle), Harness wajib menyediakan visibilitas 100% terhadap setiap keputusan tersebut. Tanpa pencatatan yang atomik, otonomi agen akan menjadi "kotak hitam" yang berbahaya. Di Makalah AI, semakin bebas agen bernalar, semakin detail sistem persistensinya mencatat.

## 2. Harness Persistence Model (`convex/schema.ts`)

### A. `harnessRuns`
Mencatat siklus hidup eksekusi utuh dari awal hingga akhir.

| Field | Tipe | Keterangan |
|---|---|---|
| `status` | `running \| paused \| completed \| failed \| aborted` | Status runtime run |
| `workflowStage` | `string` | Mirrors `paperSessions.currentStage`; default `"intake"` |
| `workflowStatus` | `running \| paused \| failed \| completed \| aborted` | Terpisah dari `status` run — workflow bisa complete saat run masih running |
| `ownerToken` | `string` | **Dua fungsi**: (1) mencegah duplicate-start race condition, (2) anchor verifikasi saat resume |
| `stepNumber` | `number` | Monotonic counter per-run; diincrement atomik via `startStepAtomic` |
| `currentStepId` | `Id<harnessRunSteps>?` | Pointer ke step yang sedang berjalan |
| `pendingDecisionId` | `string?` | **Block marker** saat run di-pause menunggu User — ini field otoratif untuk pause/resume |
| `policyState` | object? | Snapshot kebijakan: `approvalMode` (`default \| required_for_high_impact`), `currentBoundary` (`read_only \| bounded_mutation \| blocked`), `lastPolicyReason` |
| `failureClass` | enum? | Klasifikasi kegagalan: `entry_failure`, `state_failure`, `tool_failure`, `verification_failure`, `guard_failure`, `unexpected_failure` |

Indeks: `by_conversation`, `by_paperSession`, `by_ownerToken`, `by_user_updated`.

### B. `harnessRunSteps`
Mencatat setiap iterasi individual dalam siklus **Thought-Action-Observation**.

| Field | Tipe | Keterangan |
|---|---|---|
| `stepIndex` | `number` | Monotonic per-run; matches `stepNumber` saat creation |
| `status` | `running \| completed \| failed` | Status step |
| `executorResultSummary` | object? | Token usage: `finishReason`, `inputTokens`, `outputTokens`, `totalTokens` |
| `verificationSummary` | object? | Hasil `verify-step-outcome`: `canContinue`, `mustPause`, `canComplete`, `completionBlockers[]`, `leakageDetected`, `artifactChainComplete`, `planComplete`, `streamContentOverridden` |
| `toolCalls` | array | Per tool: `toolName`, `toolCallId`, `resultStatus` |

### C. `harnessEvents`
Log *append-only* yang mencatat setiap peristiwa teknis.

| Field | Keterangan |
|---|---|
| `eventId` | Stable UUID dedupable; generated by mutation if omitted |
| `eventType` | Canonical string dari `event-types.ts` (29 tipe terdaftar) |
| `correlationId` | Groups semua event untuk **satu tindakan User** — `requestId` dari `RunLane` |
| `causationEventId` | Pointer ke event penyebab dalam causal chain |
| `runId` | Optional — absent untuk pre-run entry events |

**29 Event Types Terdaftar** (dari `event-types.ts`):

| Kategori | Event Types |
|---|---|
| Entry (3) | `user_message_received`, `user_tool_result_received`, `user_decision_received` |
| Run Lifecycle (6) | `run_started`, `run_resumed`, `run_paused`, `run_completed`, `run_failed`, `run_aborted` |
| Step Execution (4) | `step_started`, `instruction_stack_resolved`, `agent_output_received`, `step_completed` |
| Tool & Mutation (8) | `tool_called`, `approval_requested`, `approval_resolved`, `execution_boundary_evaluated`, `tool_result_received`, `artifact_mutation_requested`, `artifact_mutation_applied`, `artifact_mutation_rejected` |
| Workflow & Verification (5) | `workflow_transition_requested`, `workflow_transition_applied`, `workflow_transition_rejected`, `verification_started`, `verification_completed` |
| User Decision (3) | `user_decision_requested`, `user_decision_resolved`, `user_decision_declined` |

Validasi kanonikal via `isHarnessEventType()` di adapter layer — Convex mutation sendiri menerima string arbitrary agar future event types bisa ditambah tanpa schema migration.

### D. `harnessDecisions`
Setiap kali agen butuh campur tangan User, sistem mencatatnya sebagai **Decision**.

| Field | Tipe | Keterangan |
|---|---|---|
| `type` | `clarification \| approval \| selection` | Jenis keputusan yang diminta |
| `status` | `pending \| resolved \| declined \| invalidated` | Status resolusi decision |
| `blocking` | `boolean` | Jika `true`, run menunggu (`pauseRun`) hingga decision ini diselesaikan |
| `workflowStage` | `string` | Stage saat decision dibuat |
| `prompt` | object | `title?`, `question`, `options[]?` (label, description?, recommended?) |

> **Catatan**: `decisionEpoch` (monotonic counter untuk mencegah race pada cancel-decision) adalah field di tabel `paperSessions`, **bukan** di `harnessDecisions`.

## 3. RunStore — Adapter Layer (`run-store.ts`)

`RunStore` adalah thin wrapper 1:1 ke Convex mutations. Tidak ada business logic; tidak ada try/catch — Convex errors propagate unchanged.

**Atomicity notes (faktual dari kode):**
- `startStep` → single atomic mutation `startStepAtomic` (increment + insert + setCurrent dalam satu Convex transaction).
- `pauseRun` → **DUA mutasi terpisah**: (1) `createDecision` → dapat `decisionId`, (2) `pauseRun` mutation. Partial failure meninggalkan pending decision tanpa run state change — retry aman karena `decisionId` stabil.
- `resumeRun` → **DUA mutasi terpisah**: (1) `resolveDecision`, (2) `resumeRun`.

## 4. Reasoning Trace & Transparency

Harness mentransformasikan data teknis mentah menjadi informasi yang bermakna bagi User via field `messages.reasoningTrace` (schema di `convex/schema.ts` L145):

| Field | Tipe | Keterangan |
|---|---|---|
| `version` | `number` | Schema version untuk evolusi payload |
| `headline` | `string` | Ringkasan singkat tindakan agen |
| `traceMode` | `curated \| transparent` | Mode presentasi — curated = simplified, transparent = raw |
| `completedAt` | `number` | Timestamp selesai |
| `durationSeconds` | `number?` | Waktu eksekusi |
| `rawReasoning` | `string?` | Raw reasoning text (hanya pada mode `transparent`) |
| `steps[]` | array | Per step: `stepKey`, `label`, `status` (`pending \| running \| done \| skipped \| error`), `progress?`, `ts`, `thought?`, `meta?` |
