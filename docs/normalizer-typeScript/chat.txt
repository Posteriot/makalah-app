Iya, itu fokus yang jauh lebih sehat. Dan menurut gue ini memang arah terbaik
  buat makalahapp sekarang: bukan menambah tool baru, tapi mengonsolidasikan
  normalizer TS yang sudah ada jadi satu ingestion normalization layer.

  Masalah intinya sekarang bukan kekurangan kemampuan, tapi kekurangan boundary.
  Logic normalisasi sudah ada, tapi tersebar di beberapa tempat dengan tanggung
  jawab yang campur:

  - sitasi dan URL dibersihkan di src/lib/citations/normalizer.ts, src/lib/
    citations/apaWeb.ts, dan src/lib/citations/url-validation.ts
  - fetch web diparse di src/lib/ai/web-search/content-fetcher.ts:1
  - file tertentu punya formatter sendiri, misalnya src/lib/file-extraction/
    xlsx-extractor.ts
  - lalu semua langsung dilempar ke chunking di src/lib/ai/chunking.ts:13 dan
    ingest di src/lib/ai/rag-ingest.ts:21

  Itu workable, tapi arsitekturnya belum jelas. Efek sampingnya:

  - susah tahu “teks final yang masuk embedding” dibentuk di mana
  - susah bikin strategy per source type
  - susah nambah cleanup lanjutan tanpa nyebar util baru ke banyak file
  - susah ukur dampak normalisasi ke kualitas retrieval

  Rekomendasi terbaik gue:

  - bikin satu modul baru, misalnya src/lib/ingestion/source-normalizer.ts
  - modul ini jadi pintu tunggal sebelum chunkContent() dan ingestToRag()
  - dia tidak menggantikan helper lama, tapi mengorkestrasi helper yang sudah
    ada

  Bentuk kontrak yang menurut gue paling waras:

  type SourceKind =
    | "html"
    | "pdf_text"
    | "docx_text"
    | "xlsx_markdown"
    | "plain_text"

  interface NormalizeSourceInput {
    sourceKind: SourceKind
    rawContent: string
    sourceUrl?: string
    title?: string
    metadata?: Record<string, unknown>
  }

  interface NormalizeSourceOutput {
    normalizedText: string
    normalizationMeta: {
      sourceKind: SourceKind
      originalLength: number
      normalizedLength: number
      appliedSteps: string[]
      warnings?: string[]
    }
    derivedArtifacts?: {
      references?: Array<{ url: string; title?: string }>
      sections?: Array<{ heading?: string; length: number }>
    }
  }

  Lalu strategy-nya eksplisit:

  - html: cleanup readability/turndown result, dedup paragraph, trim
    boilerplate, heading repair
  - pdf_text: normalize whitespace, dedup repeated header/footer, collapse
    broken line wraps
  - docx_text: rapihin paragraph break dan heading-ish lines
  - xlsx_markdown: pakai output existing formatter lalu kasih cleanup ringan
  - plain_text: normalization minimal

  Yang penting di sini: chunking jangan lagi menerima “content mentah dari mana
  saja”, tapi menerima content yang sudah dinormalisasi. Jadi boundary-nya jadi
  jelas:

  - extract/fetch
  - normalize
  - chunk
  - embed
  - ingest

  Kalau lo mau versi yang paling pragmatis, gue sarankan fase implementasi
  begini:

  1. Bungkus dulu jalur web source saja.
  2. Setelah stabil, baru masukkan pdf/docx/xlsx.
  3. Tambahkan telemetry sederhana: panjang sebelum/sesudah, langkah yang
     dipakai, dan source kind.

  Ini memperingan, bukan menambah beban, karena:

  - logic lama tetap dipakai
  - yang berubah cuma arsitektur alurnya jadi rapi
  - debugging jadi lebih gampang
  - nanti kalau mau eksperimen apa pun, termasuk algoritme cleanup lain, titik
    masuknya sudah jelas

  Kalau mau lanjut, pertanyaan paling penting: lo mau normalization layer ini
  khusus untuk RAG ingestion saja, atau sekalian dipakai juga untuk exact-source
  context di web search compose flow?


› rekomendasimu?


• Rekomendasi gue: buat satu normalization layer khusus untuk ingestion RAG
  dulu, jangan sekalian dipakai ke exact-source context.

  Kenapa ini yang paling tepat:

  - Jalur RAG sekarang paling butuh boundary yang rapi: hasil fetch/extract
    masih langsung dekat ke src/lib/ai/chunking.ts:13 dan src/lib/ai/rag-
    ingest.ts:21.
  - Exact-source context lebih sensitif ke fidelity. Kalau lo terlalu agresif

    justru harus dipertahankan.
  - Jadi strategi paling waras: RAG pakai normalized text, tapi exact-source
    tetap pakai source text yang paling dekat ke hasil extract/fetch asli.

  Implementasi yang gue sarankan:

  - Tambah modul src/lib/ingestion/source-normalizer.ts
  - Modul ini orkestrasi helper yang sudah ada, bukan rewrite total
  - Scope fase 1: html dan plain_text/web article
  - Cleanup yang aman:
      - trim whitespace
      - dedup paragraf identik
      - rapihin heading markdown
      - buang section boilerplate yang jelas non-konten
  - Output:
      - normalizedText
      - normalizationMeta
      - opsional derivedArtifacts

  Lalu alurnya jadi:

  - fetch/extract -> normalizeSourceContent() -> chunkContent() -> embed ->
    ingestToRag()

  Yang jangan lo lakukan sekarang:

  - jangan campur exact-source dengan hasil normalisasi ini
  - jangan bikin strategy semua format sekaligus
  - jangan bawa just-bash dulu

  Kalau gue harus kasih satu keputusan tegas: konsolidasikan normalizer TS yang
  sudah ada, pakai dulu hanya untuk ingestion RAG, dan jaga exact-source tetap
  dekat ke raw source.

  Pertanyaan kuncinya: lo mau fase 1 ini masuk hanya ke web-search content, atau
  sekalian upload file text juga?


› web-search content sekaligus upload file text


• Rekomendasi gue: fase 1 langsung mencakup web-search content sekaligus upload
  file text, tapi tetap dalam satu normalization layer yang sama dan dengan
  strategy per source type. Itu opsi terbaik karena lo dapat boundary arsitektur
  yang rapi sekarang, tanpa nambah runtime/tool baru.

  Kenapa ini paling pas:

  - Dua jalur itu ujungnya sama-sama masuk ke RAG, jadi wajar kalau mereka lewat
    pintu normalisasi yang sama sebelum src/lib/ai/chunking.ts:13 dan src/lib/
    ai/rag-ingest.ts:21.
  - Logic existing sudah cukup buat jadi fondasi:
      - web fetch parse di src/lib/ai/web-search/content-fetcher.ts:92
      - PDF/DOCX/XLSX extractor di src/lib/file-extraction/pdf-extractor.ts:35,
        src/lib/file-extraction/docx-extractor.ts:35, src/lib/file-extraction/
        xlsx-extractor.ts
  - Kalau lo pecah jadi dua layer terpisah sekarang, itu malah bikin duplikasi
    rule cleanup dan susah dijaga konsisten.

  Bentuk implementasi yang gue sarankan:

  - Tambah src/lib/ingestion/source-normalizer.ts
  - Tambah src/lib/ingestion/source-normalizer.types.ts
  - Kalau perlu, pecah strategy kecil ke:
      - src/lib/ingestion/strategies/normalize-html.ts
      - src/lib/ingestion/strategies/normalize-pdf-text.ts
      - src/lib/ingestion/strategies/normalize-docx-text.ts
      - src/lib/ingestion/strategies/normalize-xlsx-markdown.ts
      - src/lib/ingestion/strategies/normalize-plain-text.ts

  Kontraknya sebaiknya satu:

  - input: sourceKind, rawContent, sourceUrl?, title?, metadata?
  - output: normalizedText, normalizationMeta, derivedArtifacts?

  Strategy awal yang realistis:

  - html: cleanup whitespace, dedup paragraf identik, repair heading markdown,

  - pdf_text: hapus repeated header/footer, gabung line wrap rusak, rapihin

  - docx_text: rapihin paragraph spacing dan heading-ish lines

  Yang jangan dilakukan:

  - jangan masukkan citation URL normalizer ke dalam text normalizer; itu domain

  - jangan bikin cleanup terlalu agresif sampai mengubah makna dokumen

  Apakah ini beban atau memperingan?

  - Memperingan secara arsitektur
  - Nyaris netral secara runtime, kalau rule awalnya ringan dan murni TS
  - Ini jauh lebih murah daripada nambah just-bash, dan jauh lebih gampang di-

  Satu hal yang perlu lo putuskan sebelum implementasi: untuk upload file, lo

› rekomendasimu?


• Rekomendasi gue: simpan dua-duanya, raw extracted text dan normalized text,
  tapi pakai peran yang berbeda.

  Yang terbaik untuk makalahapp:

  - raw extracted text disimpan untuk audit, debugging, dan fallback
  - normalized text dipakai untuk chunking, embedding, dan retrieval

  Kenapa ini paling tepat:

  - Kalau hasil retrieval jelek, lo bisa cek apakah masalahnya ada di extractor
    atau di normalizer
  - Kalau normalizer salah buang bagian penting, lo masih punya sumber mentah
  - Lo bisa bandingin kualitas sebelum/sesudah normalisasi tanpa re-extract file
  - Ini bikin pipeline lebih aman untuk evolusi rule ke depan

  Tapi jangan bodoh juga: jangan jadikan keduanya sama-sama jalur aktif untuk
  RAG. Itu cuma bikin index ganda dan noise.

  Jadi keputusan tegasnya:

  - Simpan raw extracted text sebagai debug/audit artifact
  - Simpan normalized text sebagai ingestion-ready content
  - Hanya normalized text yang masuk ke src/lib/ai/chunking.ts:13 dan src/lib/
    ai/rag-ingest.ts:21

  Kalau mau lebih efisien:

  - Untuk file kecil/normal: simpan full raw + full normalized
  - Untuk file sangat besar: raw bisa dicap atau dipotong dengan metadata,
    asalkan masih cukup untuk audit

  Kalau gue yang putuskan implementasinya:

  - rawContent
  - normalizedContent
  - normalizationMeta
  - normalizerVersion

  Itu struktur paling waras. Tanpa itu, nanti lo susah bedain bug extractor vs
  bug normalizer.
