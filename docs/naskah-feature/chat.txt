 
 › Fokus ke halaman chat, ke chat window dan artifact, serta validation panel. Aku ingin kau memikirkan kelayakan dan potensi implementasi ide ini: 1.
  Aplikasi ini ditambah dengan fitur "dokumen utama", dengan tampilan dan cara kerja seperti ms word atau google docs. 2. Dokumen utama ini adalah kumpulan
  (copy paste) dari konten yang ada di artifak yang sudah tervalidasi atau final. 3. User bisa menggunakannya untuk melihat keseluruhan dari paper yang
  diprosesnya. Sehingga dokumen utama ini menjadi "paper jadi, dengan format A4 dan spasi 1,5, sehingga user sudah tahu, berapa halaman yang dihasilkannya.
  4. Di dokumen utama ini user bisa mengedit manual, memperbaiki format, yang hasilnya dia bisa download sebagai pdf atau docx.
  
 ---
 
 Verdict

  Ide ini layak dan punya nilai produk yang kuat. Rekomendasi terbaik gue: bikin dokumen utama sebagai dokumen terpisah yang editable, lalu isi awalnya di-
  compile dari artifact/stage yang sudah tervalidasi. Jangan jadikan artifact yang ada sekarang sebagai dokumen final langsung, dan jangan bikin
  sinkronisasi dua arah dulu.

  Alasannya sederhana: struktur sekarang belum didesain buat satu naskah komposit. ArtifactPanel dan ArtifactViewer masih fokus ke satu artifact per tab,
  bukan satu paper gabungan lintas stage di src/components/chat/ArtifactPanel.tsx dan src/components/chat/ArtifactViewer.tsx. Workflow validasi juga masih
  stage-based, dengan satu artifactId per stage di convex/schema.ts dan approval panel yang nongol di footer chat di src/components/chat/ChatWindow.tsx plus
  src/components/paper/PaperValidationPanel.tsx.

  Kenapa feasible

  Fondasinya sebenarnya udah ada:

  - Editor library sudah tersedia lewat TipTap di package.json.
  - Export server-side DOCX dan PDF A4 juga sudah ada di src/app/api/export/word/route.ts, src/app/api/export/pdf/route.ts, src/lib/export/word-builder.ts,
    dan src/lib/export/pdf-builder.ts.
  - Compiler paper dari stageData juga udah ada di src/lib/export/content-compiler.ts.

  Tapi ada gap besar:

  - Export sekarang cuma valid untuk session completed, jadi belum cocok untuk “paper jadi sementara” selama proses berjalan, lihat src/lib/export/
    validation.ts.
  - Tombol DOCX/PDF di artifact viewer sekarang masih palsu secara praktis, karena dia masih download .md, bukan export beneran, di src/components/chat/
    ArtifactViewer.tsx.
  - Compiler export sekarang ambil dari stageData, bukan dari hasil manual edit user di dokumen utama. Jadi kalau user ngedit manual, perubahan itu belum
    punya tempat penyimpanan yang cocok.

  3 pendekatan

  1. Compiled preview only
     Rebuild dokumen utama dari stage tervalidasi setiap kali dibuka. Cepat, aman, tapi user nggak bisa edit manual permanen.
  2. Editable main document, one-way sync from approved stages REKOMENDASI TERBAIK
     Saat stage disetujui, kontennya masuk ke dokumen utama sebagai draft terstruktur. Setelah itu user boleh edit manual. Kalau stage direvisi lagi,
     tampilkan “update tersedia” per section, bukan auto-overwrite.
  3. Bi-directional sync

  Rekomendasi implementasi

  Gue rekomendasiin fase seperti ini:

  - Fase 1: tambah mainDocument entity baru di Convex, terhubung ke paperSession.
  - Fase 2: saat stage approve, salin hasil final stage ke section yang sesuai di dokumen utama.
  - Fase 3: tampilkan tab/panel baru di area artifact sebagai Dokumen Utama, dengan editor paged A4.
  - Fase 4: export PDF/DOCX dari mainDocument, bukan lagi dari compiler completed-only.
  - Fase 5: kalau ada revisi stage setelah user sudah edit manual, kasih status section out of sync dan tombol review merge.

  Kalau lo pakai pola ini, dampaknya ke UX juga paling rapi:

  - Chat window tetap jadi tempat diskusi dan approval.
  - Validation panel tetap fokus ke approve/revisi per stage, bukan ngurus formatting akhir.
  - Artifact tetap jadi output per tahap.
  - Dokumen utama jadi lapisan final assembly dan editing.

  Risiko utama

  - Konflik sinkronisasi: user edit manual, lalu stage lama direvisi.
  - Granularitas section: artifact saat ini belum tentu selalu punya struktur yang bersih untuk dipetakan otomatis ke dokumen utama.
  - Preview halaman A4 di web sering “mirip Word”, tapi tidak identik 100%. Jadi page count harus dianggap estimasi UI sampai export final.
  - Kalau lo mau full Word-like editing, effort frontend-nya lumayan, walau dependensi dasarnya sudah ada.

  Kesimpulan

  Secara produk, ini ide yang sangat masuk akal karena menjawab gap paling nyata: user belum punya “paper utuh” selama proses berlangsung. Secara teknis,

  Jawaban lo ke pertanyaan ini bakal nentuin desain detailnya: dokumen utama mau jadi sumber final mandiri setelah user edit, atau tetap harus selalu tunduk
  ke artifact/stage sebagai sumber kebenaran? Kalau lo jawab itu, gue bisa lanjut bikin desain arsitektur yang lebih presisi untuk chat window, artifact
  area, validation panel, dan model datanya.


› kalau begini berarti ketika dokumen utama diedit, maka tidak compliant dengan artifact ya?


• Iya, kalau pakai rekomendasi gue tadi, begitu dokumen utama diedit manual, dia memang tidak lagi 100% compliant dengan artifact sumber.

  Tapi justru itu keputusan desain yang paling waras. Kalau lo paksa tetap selalu compliant penuh, berarti ada dua kemungkinan, dan dua-duanya jelek:

  - Edit manual user harus dilarang atau sangat dibatasi.
  - Setiap perubahan di dokumen utama harus balik nyinkron ke artifact/stage, dan itu bikin conflict logic jadi berat banget.

  Jadi yang paling masuk akal itu bukan pura-pura “tetap compliant”, tapi compliance-nya dibuat eksplisit:

  - Artifact = source of truth per tahap.
  - Dokumen utama = compiled manuscript + manual edits user.
  - Sistem kasih status per section: synced, edited manually, outdated karena artifact berubah.

  Dengan model ini, user tetap bebas edit, tapi sistem jujur soal status sinkronisasinya. Menurut gue ini opsi terbaik, karena Word/Docs-style editing
  memang secara alami menghasilkan divergensi dari source modularnya.

  Kalau lo mau, gue bisa lanjut bikin 3 model sinkronisasi yang lebih tegas:

  1. strict compliance
  2. manual override

  Satu pertanyaan kunci: lo maunya setelah user edit dokumen utama, artifact tetap dianggap referensi teknis saja, atau lo mau ada tombol apply back to
  artifact untuk section tertentu?


› Kalau apply back akan sangat rumit. Justru kalau artifact yang diedit (yang sekarang sudah bisa) , dokumen utama menyesuaikan akan lebih mudah tho?


• Iya, itu jauh lebih masuk akal.

  Kalau arah sinkronisasinya artifact -> dokumen utama, kompleksitasnya turun banyak, karena:

  - artifact per stage memang sudah jadi unit yang jelas dan sekarang memang sudah bisa diedit di src/components/chat/ArtifactViewer.tsx.
  - paperSession.stageData.<stage>.artifactId sudah nyimpen relasi satu stage ke satu artifact aktif, jadi sistem punya anchor yang stabil buat recompile
    dokumen utama dari artifact final per tahap.
  - Lo nggak perlu bikin sinkronisasi balik dari dokumen utama ke artifact, yang itu memang ruwet.

  Jadi model yang paling kuat menurut gue:

  - Artifact tetap jadi source of truth per section/tahap.
  - Dokumen utama jadi compiled view dari artifact-artifact yang approved/final.
  - Kalau user edit artifact, dokumen utama auto ikut update.
  - Kalau user edit dokumen utama, itu harus dianggap override lokal dan statusnya berubah, misalnya manual override atau detached.

  Masalahnya justru kalau lo mau dua-duanya editable penuh dan tetap sinkron otomatis. Itu yang bakal bikin conflict. Jadi rekomendasi terbaik gue:

  - Untuk fase awal, izinkan edit penuh di artifact.
  - Dokumen utama awalnya read-only compiled A4 preview + export.
  - Edit manual di dokumen utama baru dibuka di fase berikutnya, dengan status jelas bahwa section itu tidak lagi sinkron.

  Itu jauh lebih feasible dan tetap bersih secara arsitektur.

  Pertanyaan kuncinya: lo mau fase pertama dokumen utama dibuat dulu sebagai compiled preview + export, tanpa edit manual di sana, supaya sinkronisasi
  dasarnya beres dulu?


› jadi dokumen utama jangan editable? hanya berfungsi, melihat tampilan utuh paper, dan berapa banyak halaman yang dihasilkan? termasuk format (markdown
  renderer) yang compliant dengan akdemic format umumnya di dokumen utama itu?


• Iya, itu justru arah yang paling tepat untuk fase awal.

  Rekomendasi terbaik gue: dokumen utama jangan editable dulu. Fungsinya:

  - menampilkan paper utuh hasil kompilasi dari artifact final/tervalidasi,
  - memakai layout A4 dengan margin dan spasi akademik yang konsisten,
  - memperlihatkan estimasi/pagination jumlah halaman,
  - jadi basis export PDF/DOCX.

  Itu jauh lebih bersih daripada langsung bikin editable. Kenapa:

  - source of truth tetap jelas di artifact per tahap,
  - dokumen utama selalu compliant dengan artifact,
  - sinkronisasi jadi satu arah dan sederhana,
  Tapi gue perlu lurusin satu hal penting: kalau lo bilang markdown renderer yang compliant dengan format akademik, itu jangan dipahami sebagai markdown
  renderer biasa. Markdown renderer standar tidak cukup untuk compliance akademik. Yang lo butuhin itu:

  - renderer paginated A4 preview,
  - style rules khusus akademik,
  - mapping heading, paragraf, daftar pustaka, tabel, kutipan, dan spacing ke format paper.

  Jadi bentuk teknis yang benar bukan:

  - “render markdown apa adanya”

  Tapi:

  - “render compiled academic document dari artifact content, dengan aturan tampilan paper”

  Dengan codebase sekarang, ini realistis karena:

  - PDF builder A4 sudah ada di src/lib/export/pdf-builder.ts
  - Word export juga sudah ada di src/lib/export/word-builder.ts
  - compiler paper dari session sudah ada di src/lib/export/content-compiler.ts

  Jadi menurut gue fase 1 yang paling kuat:

  - artifact tetap tempat edit
  - dokumen utama read-only
  - dokumen utama pakai tampilan paper akademik A4, bukan panel artifact biasa
  - dokumen utama auto-update saat artifact final berubah
  - dokumen utama bisa diexport

  Satu catatan penting: jumlah halaman di preview web sebaiknya dianggap preview yang sangat dekat, bukan jaminan absolut 100% sama dengan Word/PDF, kecuali
  kita paksa preview engine-nya mengikuti aturan export dengan sangat ketat. Itu bisa dikejar, tapi jangan diasumsikan gratis.

  Pertanyaan kunci gue: lo mau dokumen utama ini muncul sebagai tab baru di area artifact, atau lo maunya dia jadi mode/workspace terpisah dari artifact
  biasa?


› Aku maunya jadi  mode/workspace terpisah dari artifact biasa
• Itu pilihan yang lebih tepat. Gue rekomendasiin dokumen utama jadi workspace terpisah, bukan tab di artifact panel.

  Alasannya kuat:

  - Artifact workspace fungsinya tetap modular: ngerjain, review, edit per bagian.
  - Dokumen utama workspace fungsinya holistik: lihat paper utuh dalam format akademik A4.
  - Secara mental model, user nggak bingung antara “blok konten per tahap” vs “naskah final utuh”.
  - Secara teknis, ini juga lebih bersih daripada maksa ArtifactPanel yang sekarang single-document oriented jadi viewer paper penuh.

  Struktur yang paling masuk akal:

  - Chat window tetap di kiri/tengah seperti sekarang.
  - Artifact workspace tetap untuk artifact per stage.
  - Dokumen utama workspace jadi area terpisah yang bisa dibuka sebagai mode khusus, berisi:
      - preview paper utuh,
      - pagination A4,
      - style akademik,
      - jumlah halaman,
      - tombol export PDF/DOCX.

  Dengan model ini, source of truth tetap jelas:

  - artifact final/validated = sumber isi
  - dokumen utama = compiled read-only presentation

  Jadi iya, arah terbaik sekarang:

  - dokumen utama non-editable
  - workspace terpisah
  - auto-refresh saat artifact stage yang relevan berubah
  - academic preview, bukan markdown viewer biasa

  Kalau gitu, pertanyaan paling penting berikutnya: lo mau dokumen utama workspace ini bisa dibuka berdampingan dengan chat, atau menggantikan artifact
  workspace saat mode itu aktif?
